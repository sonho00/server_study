## Troubleshooting:타임존 연산 오버헤드로 인한 비동기 I/O 핸들 파괴 분석

<br>

### 1. 문제 상황

현상: 로그 출력 포맷을 개선한 후, IOCP 서버의 `WSARecv` 호출 단계에서 `ERROR_PATH_NOT_FOUND (ErrorCode 3)` 에러가 발생함.

영향: 클라이언트의 접속(`Accept`)은 정상적으로 이루어지나, `GQCS` 이후 첫 데이터를 수신하기 직전에 소켓 통로가 파괴되며 연결이 강제 종료됨.

<br>

### 2. 디버깅 과정

#### 2.1 고정관념에 의한 오판

초기 대응: 발생한 에러 코드가 커널 레벨의 핸들 에러(`ErrorCode 3`)였기 때문에, 단순한 로그 출력 함수가 원인일 것이라고는 전혀 상상하지 못함.

시행착오 및 지체 원인:

소켓 API 수준의 검토: `ERROR_PATH_NOT_FOUND`라는 에러 메시지에 매몰되어,
소켓 생성 시점부터 `AcceptEx` 호출 이후의 `SO_UPDATE_ACCEPT_CONTEXT` 옵션 적용까지 소켓 API의 호출 순서와 파라미터를 중점적으로 검토함.

엔진 내부 로직 분석: 소켓 호출부에 결함이 발견되지 않음에도 로그는 로직에 영향이 없을 것이라는 고정관념에 사로잡혀,
세션 관리나 핸들링 로직 등 엔진 내부의 복잡한 상태 관리 코드를 분석하며 상당한 시간을 소모함.

<br>

#### 2.2 가설의 전환과 `git bisect` 도입

현재 로직에 문제가 없다면, 이전 커밋 어딘가에 잠재되어 있던 설계 결함이 환경 변화(로그 수정 등)를 만나 터진 것이라 가설을 수정함.

잠재적 버그가 시작된 지점을 찾기 위해 과거 커밋들을 대상으로 `git bisect`를 실행하여 이분 탐색을 진행함.

결과: bisect가 지목한 첫 번째 불량 커밋(First Bad Commit)은 아래와 같음.

Bad Commit Hash: [`f35dfca`](https://github.com/sonho00/server_study/commit/f35dfcaa3ded9264777df9c311a275f6a4c53424) (Refactor: 단계별 로그 시스템 구축 및 인터페이스 표준화)

<br>

#### 2.3 지점 특정

도구의 결과를 바탕으로 편견을 버리고 로그 코드를 분석함. `GQCS` 직후 호출되는 `LOG_DEBUG`를 주석 처리했을 때 에러가 사라지는 것을 확인하고 최종 원인을 확정함.

<br>

### 3. 원인 분석

#### 3.1 `std::chrono::current_zone()`의 오버헤드

로컬 시간을 구하기 위해 사용한 `std::chrono::current_zone()`은 내부적으로 시스템 타임존 데이터베이스를 조회하며,
이 과정에서 전역 락을 점유하거나 예기치 못한 지연을 유발함.

<br>

#### 3.2 `Critical Path` 지연과 OS 핸들 무효화

`Hot Path` 방해: `GQCS`로 신호를 받은 직후는 해당 세션의 다음 I/O를 즉시 예약해야 하는 매우 민감한 타이밍임.

에러 발생: 로그 함수가 CPU를 점유하는 동안 유저 모드 스레드가 지연되었고, 그 사이 커널은 해당 소켓의 통로가 유효하지 않다고 판단하여 파괴함.
이로 인해 뒤늦게 호출된 `WSARecv`가 `ERROR_PATH_NOT_FOUND`를 반환한 것으로 파악됨.

<br>

### 4. 해결 방법

코드 수정: 무거운 `std::chrono::current_zone()` 호출을 제거하고, 오버헤드가 거의 없는 `std::chrono::system_clock` 기반의 가벼운 타임스탬프 로직으로 교체함.

Fix Commit Hash: [`e518947`](https://github.com/sonho00/server_study/commit/e518947fbf8a52d6e39f29256f7dc9245602b244) (Fix: 로그 출력 지연으로 인한 `IOCP` 소켓 핸들 파괴 오류 수정)

추가 검증: 동일한 코드를 부하가 적은 초기화 구간(`Main`)에서 실행했을 때는 문제가 없음을 확인. 이는 함수 자체가 틀린 것이 아니라 사용된 위치가 문제였음을 입증함.

<br>

### 5. 교훈

##### 잘 모르는 코드는 무조건 의심하라
AI가 제안했거나 표준 라이브러리에 포함된 기능이라 할지라도,
내부 동작 원리와 비용을 정확히 파악하지 못했다면 고성능 환경에서는 언제든 시스템을 파괴하는 원인이 될 수 있다.

##### 고정관념의 위험성
로그는 로직에 영향이 없다는 믿음이 디버깅 시야를 좁혔음.
고성능 서버 환경에서는 시스템을 구성하는 모든 요소가 실행 흐름의 일부임을 인지해야 한다.

##### 데이터와 도구 기반의 디버깅
주관적인 직감으로 시간을 낭비하던 상황을 끝낸 것은 결국 `git bisect`라는 객관적인 데이터였음.
내 직감이 도구의 결과와 충돌할 때 도구를 믿는 것이 문제를 해결하는 가장 빠른 길임을 배움.
